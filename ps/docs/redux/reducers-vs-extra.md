# `reducers` vs `extraReducers` in Redux Toolkit

| Aspect | `reducers` | `extraReducers` |
| --- | --- | --- |
| Purpose | Handle actions generated by the slice itself | Respond to actions outside the slice (e.g., thunks, other slices) |
| Definition | Object of reducer functions keyed by action name | Builder callback (`builder.addCase`) or object notation |
| Action creators | Auto-generated (`slice.actions`) | None generated; actions must exist elsewhere |
| Typical use | Local sync updates (`increment`, `reset`) | Async lifecycles (`pending`, `fulfilled`, `rejected`) |

## Example
```js
const profileSlice = createSlice({
  name: 'profile',
  initialState: { status: 'idle', data: null },
  reducers: {
    reset(state) {
      state.status = 'idle';
      state.data = null;
    }
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchProfile.pending, (state) => {
        state.status = 'loading';
      })
      .addCase(fetchProfile.fulfilled, (state, action) => {
        state.status = 'succeeded';
        state.data = action.payload;
      });
  }
});
```

## Guidance
1. Place local, synchronous mutations in `reducers` for discoverability and auto-generated action creators.
2. Use `extraReducers` for shared actions or `createAsyncThunk` lifecycles to avoid circular imports.
3. Keep builder chains readable by grouping related thunks and using `addMatcher` for shared patterns.
